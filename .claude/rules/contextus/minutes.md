# 議事録（Meeting Minutes）— 規約

## 概念

セッション（会話）全体が一つの「会議」である。
議事録は決定のリストではなく、**会議の全体の記録**：
- 誰が何を言ったか
- どのように議論が展開したか
- どのような結論に至ったか

コーディングセッションでも知識作業セッションでも同じ概念が適用される。

## 保存先

`.spec/MINUTES.md`（プロジェクトリポジトリ内）

`.spec/` ディレクトリがない場合は `MINUTES.md`（プロジェクトルート）。

## フォーマット

```markdown
# 議事録 — <project-name>

## 会議: YYYY-MM-DD HH:MM
**出席者**: <人間名>（人間）、<エージェント名>（エージェント）
**場所**: <プロジェクト名> セッション

---

### 議題N: <話題のタイトル>
**HH:MM〜HH:MM**

<誰が何を言い、議論がどう展開したかの要点>

**決定**: <結論>（なければ省略）
**実施**: <実際に行ったこと>（なければ省略）

---
```

## いつ書くか

「決定の瞬間を待つ」のではなく、**会議を録音するように継続的に書く**。

| タイミング | 行動 |
|---|---|
| セッション開始時 | 会議ヘッダー（出席者・日時・場所）を書く |
| 新しい話題が始まるとき | 議題セクションを開く |
| 話題が展開するにつれて | 対話の要点を追記する |
| 話題に区切りがついたとき | 決定・実施事項を記録して閉じる |
| セッション終了時 | 未完の議題があれば「継続中」と記録する |

## 議題のタグ（任意）

話題の種類を簡潔に示すタグを議題タイトルに付けてよい:
- `[調査]` — 調査・評価
- `[設計]` — 設計・アーキテクチャ議論
- `[実装]` — 実装作業
- `[レビュー]` — コードレビュー・設計レビュー
- `[計画]` — 計画立案・優先順位付け

## クラッシュ耐性への注意

議事録はエージェントがテキストとして書くため、
プロセスのクラッシュ時に最後の書き込みが不完全になる可能性がある。

重要な議論の後は git コミットすることを推奨:
```bash
git add .spec/MINUTES.md && git commit -m "docs: update MINUTES.md"
```
