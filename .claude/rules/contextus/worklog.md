# 作業ログ（Work Log）— 規約

## 概念

タスク単位で「何のために・何をしたか・結果はどうだったか」を記録する。
git log は「何をコミットしたか」を記録するが、
作業ログは「なぜそれをしたか・何を試みたか・何を学んだか」を記録する。

コーディングセッションでも知識作業セッションでも同じ概念が適用される。

## 保存先

`.spec/WORKLOG.md`（プロジェクトリポジトリ内）

`.spec/` ディレクトリがない場合は `WORKLOG.md`（プロジェクトルート）。

## フォーマット

```markdown
# 作業ログ — <project-name>

## YYYY-MM-DD

### [HH:MM-HH:MM] <タスクの名前>
**目的**: <なぜこのタスクをするのか>
**実行**:
- <実際に行ったこと（箇条書き）>
- <試みたがうまくいかなかったことも含める>
**結果**: ✅/🔄/❌ <結論・成果>（commitハッシュがあれば記載）

---
```

## いつ書くか

| タイミング | 行動 |
|---|---|
| タスクを開始するとき | タスクセクションを開き、「目的」を書く |
| タスクを実行するにつれて | 実行した内容を追記する |
| タスクが完了したとき | 「結果」を書いて閉じる |
| セッション終了時 | 未完タスクを🔄として記録する |

## タスクの単位

「タスク」は厳密な定義はないが、概ね以下の粒度:
- 一つのまとまった目的を持つ作業（調査、設計、実装、レビューなど）
- 30分〜2時間程度の作業単位
- git コミットが 1〜数個対応する範囲

## git 情報との統合

作業ログには git の情報を含めるとより有用:

```markdown
**結果**: ✅ 完了（commit: abc1234 docs: update KNOWLEDGE.md）
```

セッション終了時に自動生成ツールが以下を追記してもよい:
```markdown
## Git Summary (YYYY-MM-DD)
### Commits
- abc1234 docs: update KNOWLEDGE.md
### Changed Files
 .spec/KNOWLEDGE.md | 30 +++
```

## クラッシュ耐性への注意

作業ログはセッション終了時に自動生成される場合、
プロセスが突然終了すると当該セッションの記録が失われる可能性がある。

重要なタスクが完了したら、エージェントが都度追記してコミットすることで耐性が上がる:
```bash
git add .spec/WORKLOG.md && git commit -m "docs: update WORKLOG.md"
```
